                RequestStage、ReqTimeStats 与时间线证据#

                这章解决什么问题#

                扩展与调试这一节已经讲了 metrics、trace、exporter、watchdog 和 crash dump，但还有一层非常关键的“统一时间语义”没有单独成章：同一条请求在系统内部经过这么多阶段，时间戳究竟是怎么被记录、切成 stage、再变成 queue time、per-stage latency 和 trace slice 的？

                这一章专门解释 req_time_stats.py 这套时间线证据体系。

                为什么这一层比普通 trace 更值得单独讲#

                因为它不是单纯“打点多一点”。ReqTimeStats 和 RequestStage 实际上做了三件事：

                • 给请求定义统一阶段名
                • 把这些阶段同时接到 metrics 和 trace 上
                • 让请求在 lifecycle、scheduler、execution、disaggregation 路径之间共享一套时间语义

                这意味着它是整本书里少数能把前文很多章节重新接到一起的“证据总线”。

                一张图：一条请求怎样被切成阶段切片#

                这张图解决的理解障碍是：读者容易把 queue time、TTFT、decode loop、dispatch time 当成彼此独立的指标，而不是同一条时间线的切片。

                flowchart LR
    Create["created"] --> Tok["TOKENIZE"]
    Tok --> APID["API_SERVER_DISPATCH"]
    APID --> Wait["PREFILL_WAITING / DECODE_WAITING"]
    Wait --> Fwd["PREFILL_FORWARD / DECODE_FORWARD"]
    Fwd --> Loop["DECODE_LOOP"]
    Loop --> Resp["response_sent_to_client"]
    Resp --> Done["finished"]

                图比纯文字多解释的一点是：很多“指标名字”其实都是同一条时间线上的稳定切片。

                RequestStage 定义了什么#

                req_time_stats.py 里的 RequestStage 不只是一个枚举，而是一组带 stage_name 和 level 的配置对象。当前至少包括：

                • TOKENIZE
                • API_SERVER_DISPATCH
                • REQUEST_PROCESS
                • PREFILL_WAITING
                • PREFILL_FORWARD
                • PREFILL_CHUNKED_FORWARD
                • DECODE_WAITING
                • DECODE_FORWARD
                • DECODE_LOOP
                • 以及 disaggregation 相关的 prepare/bootstrap/transfer 阶段

                这说明系统不是临时散点打时间戳，而是在尝试维护一套跨模块统一的 stage 语义。

                ReqTimeStats 为什么是贯穿全书的隐藏主线#

                你在很多地方都会看到它：

                • TokenizerManager 在 tokenize 和 API dispatch 前后设置时间。
                • Scheduler 在入队、等待、forward、retract、schedule 时更新阶段。
                • 请求完成和 response sent to client 时又会更新末端时间。

                也就是说，虽然 ReqTimeStats 本身不像 Scheduler 那样显眼，但它是整条主链路的统一时间记录器。

                set_*_time() 方法真正构成了什么#

                这组方法最好别只当 setter 读。它们实际上定义了一套稳定的阶段语义：

                • set_created_time()
                • set_tokenize_finish_time()
                • set_api_server_dispatch_time() / finish
                • set_wait_queue_entry_time()
                • set_forward_entry_time()
                • set_prefill_finished_time()
                • set_last_decode_finish_time()
                • set_response_sent_to_client_time()
                • set_finished_time()

                这让不同层在记录时间时不需要各自发明一套名字，而是共享一套全书都能对齐的时序语言。

                metrics 和 trace 为什么都依赖它#

                ReqTimeStats 会：

                • 在某些阶段直接 observe_queue_time(...) 或 observe_per_stage_req_latency(...)
                • 在 trace 开启时，用 trace_ctx.trace_slice_start(...) / trace_slice_end(...) / trace_event(...)

                也就是说，metrics 和 trace 不是两套互不相干的证据链，而是共享同一套 stage 语义。这就是为什么前面 observability 章节能自然和这里回扣。

                response_sent_to_client_time 为什么特别重要#

                很多系统只记录 finished time，不记录真正“已经发回客户端”的时间。这里明确维护了 response_sent_to_client_time，说明 SGLang 在区分：

                • 系统内部认为请求完成
                • 客户端真正收到最终响应

                这对 streaming 和非 streaming 场景都特别关键，因为它能避免把“内部处理结束”和“外部感知完成”混成一个概念。

                retraction、disaggregation 和 fake output 为什么都进入这套时间线#

                这是 ReqTimeStats 特别像一本好书应该强调的地方。它不仅覆盖快乐路径，还覆盖：

                • set_retract_time()
                • prefill/decode bootstrap
                • transfer KV cache
                • fake output / quick finish

                这说明这套时间线不是只服务正常请求，而是试图把异常、优化和复杂部署路径也纳入同一时间语义。

                这一层最容易出现的误判#

                1. 把 queue time 只当成 scheduler 指标#

                其实 queue time 只是更长时间线里的一个切片。

                2. 以为 trace 和 metrics 的阶段定义各自独立#

                源码表明两者在很大程度上共享 RequestStage 语义。

                3. 把 finished time 当成用户感知完成时间#

                response_sent_to_client_time 的存在说明两者不能混为一谈。

                如果你要用这套时间线排障，先怎么走#

                建议按这个顺序：

                1. 先确认请求有没有完整 created -> response_sent_to_client -> finished。
                2. 再确认瓶颈主要落在 tokenize、dispatch、waiting、forward 还是 decode loop。
                3. 如果涉及 retract、disaggregation 或 quick finish，再看是否进入了特殊 stage。
                4. 最后再把这些阶段与 trace、metrics、request log 对齐。

                小结#

                这一章真正想补齐的，是整本书维护层里最通用的一条证据主线：

                • RequestStage 给不同层定义了统一阶段名。
                • ReqTimeStats 把这些阶段落成时间切片。
                • metrics、trace、response sent time 都围绕这条时间线工作。

                到这里，扩展与调试部分就不只是在列工具，而是开始给读者一条统一的“时间语义地图”。
