                Request logger、straggler 与 crash dump 证据链#

                这章解决什么问题#

                扩展与调试部分已经讲了 observability、request metrics exporter、dump 工具和 liveness，但还缺一条非常贴近实际事故调查的链：系统在请求收尾时怎样记录 request log，怎样积累定期 dump，怎样保留 crash 前后的请求材料，以及怎样把 straggler 请求从普通流量里筛出来。

                这一章专门把 RequestLogger 和 TokenizerManager 里的 dump/crash/straggler 证据链拉成一条完整路径。

                为什么这一层不该和“普通日志”混为一谈#

                RequestLogger 不是简单的 logger.info(...) 包装。它负责的是：

                • 接收请求时的结构化记录
                • 请求完成时的结构化记录
                • 与 metrics exporter 的字段筛选口径协同

                而 TokenizerManager 在它之上又叠了一层：

                • dump_request_list
                • crash_dump_request_list
                • straggler_request_list
                • dump_requests_folder
                • dump_requests_threshold

                这说明“请求日志”和“事故证据”虽然共享部分信息，但在系统里是两层不同能力。

                一张图：请求从完成到事故证据的几条分支#

                这张图解决的理解障碍是：很多读者会把 finished request 只联想到日志输出，但实际上它可能同时流向多个证据槽。

                flowchart LR
    Finish["request finished"] --> Log["RequestLogger.log_finished_request()"]
    Finish --> Metrics["request metrics exporter"]
    Finish --> Dump["dump_request_list / thresholded flush"]
    Finish --> Crash["crash_dump_request_list (recent window)"]
    Finish --> Strag["straggler_request_list"]

                图比纯文字多解释的一点是：一个请求在完成之后，并不是只写一条日志，而可能按不同调查目的进入不同证据容器。

                RequestLogger 真正提供了什么#

                utils/request_logger.py 里最重要的两个入口是：

                • log_received_request(...)
                • log_finished_request(...)

                这对全书主线特别重要，因为它们分别卡在请求生命周期的首尾两端。也就是说，request logger 不是孤立功能，而是与前面写过的 TokenizerManager.generate_request(...) 和 _wait_one_response(...) 形成回环。

                TokenizerManager 为什么要额外维护三类请求列表#

                dump_request_list#

                这更像定期批量落盘容器。系统会把达到阈值的一批请求统一 dump 到文件，而不是每完成一条就立刻碎片化写出。

                crash_dump_request_list#

                这是更偏“保留最近一段窗口内请求证据”的缓冲区。真正 crash 时，它能和未完成请求一起被打包出去，帮助你在事故后还原现场。

                straggler_request_list#

                这是把慢请求从普通请求流里单独拎出来的证据槽。它的价值不是多收一点日志，而是把“尾部异常请求”从海量正常流量里显式分层。

                这三类列表共同说明：SGLang 把请求证据分成了“常规记录”“定期快照”“事故窗口”“慢请求特写”四个调查层次。

                dump_requests_threshold 透露了什么工程判断#

                TokenizerManager 默认会维护 dump_requests_threshold = 1000。这说明系统不想对每条请求都做重型持久化，而是采用“攒到一定规模再批量 dump”的策略。收益是更低运行时干扰；代价是单条请求的即时证据可见性没有那么强。

                这正是一本好技术书应该强调的取舍：证据收集本身也有成本，因此系统会按调查目的分层，而不是一刀切。

                crash dump 的窗口语义为什么重要#

                源码里 crash_dump_request_list 会清理过旧请求，并在真正 crash 时把“最近完成的请求”和“仍未完成的请求”一起 dump。这个设计特别有价值，因为事故调查真正需要的通常不是全量历史，而是：

                • crash 前最近发生了什么
                • crash 时哪些请求还挂在系统里

                这比简单把所有历史日志都翻出来要高效得多。

                straggler 证据为什么适合单独收集#

                如果慢请求只存在于普通 finished log 里，它很容易被淹没。单独的 straggler_request_list 等于承认了一件工程事实：尾部问题值得有自己的证据通道。

                这也和前面关于 P99、streaming backlog、retract、watchdog 的章节形成了回扣。整本书开始变成一套真正互相支撑的维护体系，而不是零散技巧。

                这一层和 request metrics exporter 的边界#

                前一章讲的是结构化请求证据落盘；这一章讲的是请求日志与事故调查容器。它们的区别可以这样理解：

                • exporter 更像长期、规则化的结构化数据积累。
                • request logger / crash dump 更像值班与事故现场的证据收集。

                两者都重要，但用途不同。

                如果你要查一次真实事故，先怎么用这条链#

                建议按这个顺序：

                1. 先看 log_received_request(...) / log_finished_request(...) 有没有形成首尾闭环。
                2. 如果问题不是单条，而是批量异常，看 dump_request_list 的批量落盘内容。
                3. 如果发生 crash，看 crash_dump_request_list 对应的窗口材料。
                4. 如果是尾部慢请求，看 straggler_request_list 是否已经把异常样本收出来。
                5. 再把这些材料和 metrics exporter、trace、dump/replay 合起来交叉验证。

                这套设计的收益与代价#

                收益：

                • 请求证据按用途分层，适合不同调查场景。
                • crash 前后的窗口材料比全量历史更适合现场还原。
                • 慢请求可以被显式拎出，而不是埋在普通日志里。

                代价：

                • 证据分散在多个容器里，维护者需要知道各自用途。
                • 阈值与窗口策略意味着你不一定总能拿到“所有东西”。
                • 如果不理解这套分层，很容易错把某个容器当成全局真相。

                小结#

                这一章真正想补齐的，是“请求级证据”在维护层里的最后一层组织方式：

                • RequestLogger 负责首尾结构化记录。
                • dump/crash/straggler 容器负责不同调查场景下的证据留存。
                • 它们和 metrics exporter、trace、dump/replay 共同组成完整证据链。

                到这里，扩展与调试部分就不仅在讲“有哪些工具”，也开始讲“这些工具和证据容器怎样分工配合”。
