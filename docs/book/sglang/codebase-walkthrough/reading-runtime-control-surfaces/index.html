                读运行时控制面：logging、weights group 与 expert record#

                这章解决什么问题#

                前面的代码导读已经把：

                • 请求主线
                • 输出尾部
                • prefix cache
                • 多 worker
                • session
                • health check

                这些主要脊梁都落回了源码，但还有一类对维护者非常重要、却仍然散落在 http_server.py、tokenizer_communicator_mixin.py、scheduler.py、engine.py 与若干 util 文件里的控制面没有被正式收束：

                • configure_logging
                • init_weights_update_group / destroy_weights_update_group
                • start_expert_distribution_record / dump_expert_distribution_record
                • remote_instance_transfer_engine_info

                这些不是普通业务 API，也不是最底层 debug hack，而是运行时真正对外暴露的维护和协调动作。

                这章的目标，就是把这组运行时控制面收成一条可读的源码路径。

                为什么这层值得单独成章#

                因为技术书如果只讲：

                • 请求怎样跑
                • 模型怎样执行
                • 指标怎样看

                却不讲“维护者怎样通过正式控制面改变系统行为”，整本书最终还是会偏向架构概览，而不是能带去现场用的系统书。

                这些控制面共同回答的是：

                • 你怎样切换请求日志与 dump 行为
                • 你怎样初始化远端权重更新组
                • 你怎样打开 / 关闭 expert distribution 记录
                • 你怎样查询远端 transfer 协调信息

                也就是说，它们构成了“系统运行中怎样被控制和协同”的另一条脊梁。

                一张图：这组控制面不是平铺端点，而是三类维护动作#

                这张图解决的理解障碍是：很多读者会把这些 API 看成一串零散运维端点，但它们实际上可以分成三个稳定簇。

                flowchart TD
    Log["日志与证据控制\nconfigure_logging"] --> TM["TokenizerManager"]
    Weights["远端权重协调\nweights_update_group / transfer info"] --> Comm["Communicator / tp_worker / model_runner"]
    Experts["专家分布记录\nstart/stop/dump expert record"] --> Rec["scheduler / expert_distribution recorder"]

                图比纯文字多解释的一点是：这组控制面不是同一层级的“杂项端点”，而是三类不同的维护动作。

                第一组：configure_logging 为什么应该单独看#

                http_server.py 里有：

                • /configure_logging

                它最终落到：

                • TokenizerManager.configure_logging(...)
                • 再进入 RequestLogger.configure(...)

                这说明请求日志并不是“启动时定好就不能动”的静态配置，而是运行中可调整的正式控制面。

                从系统设计角度看，这很有价值，因为它意味着：

                • 你可以在正常运行时把请求日志从轻量模式切到更重模式
                • 然后再在问题窗口期抓证据

                也就是说，日志控制本身就属于 live control plane，而不是只属于启动参数。

                第二组：weights update group 为什么不是普通权重更新#

                前面的章节已经讲过：

                • update_weights_from_*

                但 init_weights_update_group(...) / destroy_weights_update_group(...) 代表的是另一层东西：

                • 不是一次具体权重替换
                • 而是先把一个分布式权重更新协同通道搭起来

                源码路径很清楚：

                • http_server.py 提供入口
                • TokenizerManager 通过 communicator 发下去
                • scheduler_update_weights_mixin.py
                • tp_worker.py
                • model_runner.py

                这说明这条控制链的核心不是数据本身，而是先建立一条“之后可以做分布式参数协同更新”的协作结构。

                从技术书角度看，这非常值得单独说清，因为它说明：

                • 权重更新控制面是分层的
                • 有的动作是直接更新权重
                • 有的动作是先搭协调组，再允许后续更新

                第三组：expert distribution record 为什么连接了执行与运维#

                http_server.py 里暴露了：

                • /start_expert_distribution_record
                • /stop_expert_distribution_record
                • /dump_expert_distribution_record

                而 scheduler 里真正会调用：

                • get_global_expert_distribution_recorder().start_record()
                • ...stop_record()
                • ...dump_record()

                同时模型与 MoE 路径里又会通过：

                • with_current_layer(...)
                • disable_this_region()
                • on_select_experts(...)

                不断把专家选择信息喂给 recorder。

                这说明 expert distribution record 不是一个普通观测端点，而是一条把：

                • 执行层专家路由
                • 全局 recorder
                • 对外运维端点

                接成闭环的控制链。

                为什么这组功能和 return_routed_experts 不能混为一谈#

                两者都跟专家路由有关，但职责不同：

                • return_routed_experts 更像单请求级别的旁路返回语义
                • expert distribution record 更像全局 / 长窗口的运行时记录能力

                这正是一本好系统书应当主动拆开的地方。否则读者会把它们都当成“专家路由可视化”的同类功能，实际却一个偏 request-level，一个偏 runtime-level。

                remote_instance_transfer_engine_info 为什么也应归到这一组#

                前一章已经解释了：

                • 远端 transfer 协调的 bootstrap server

                但这里更值得补一层维护者视角：

                • /remote_instance_transfer_engine_info

                是把那条远端协调链重新暴露到主控制面上的查询入口。

                把它放在这章，而不是只留在运行时架构里，是因为从维护者视角看，它更像一项“在线查询某个协调状态”的控制面动作。

                也就是说：

                • 运行时架构章节负责讲它为什么存在
                • 这章负责讲维护者怎样把它当作正式控制面来读

                这组控制面的共同特点是什么#

                它们虽然分属不同主题，但有三个共同点：

                1. 都不是普通业务请求
                2. 都会穿过正式 communicator / manager / scheduler / worker 链
                3. 都在运行中改变或查询系统的维护状态，而不是模型业务语义本身

                这说明“运行时控制面”本身就是一类稳定概念，而不只是几个刚好长得像运维接口的端点。

                如果你要顺着源码读这组控制面，推荐顺序是什么#

                建议按下面顺序：

                1. 先看 http_server.py 里的路由入口
                2. 再看 TokenizerManager / communicator mixin 的方法
                3. 然后分叉到：
                  • RequestLogger.configure(...)
                  • scheduler_update_weights_mixin.py / tp_worker.py / model_runner.py
                  • scheduler.py 里的 expert_distribution_handle(...)
                  • bootstrap / transfer info 查询路径

                这样读，你先抓住“控制请求怎样进入系统”，再看“它们分别改变哪一层”。

                这一层最容易出现的误判#

                1. 以为这些只是零散运维 API#

                它们实际上组成了一条正式运行时控制面。

                2. 以为 configure_logging 只是本地工具#

                它是运行中可切换请求日志与 dump 策略的正式入口。

                3. 以为 weights update group 和具体权重更新是一回事#

                前者更像先搭协同通道，后者才是真正更新数据。

                4. 以为 expert distribution record 和 return_routed_experts 是同一种能力#

                一个偏全局 recorder，一个偏单请求旁路返回。

                小结#

                这一章真正要补齐的，是代码导读里此前散落的运行时控制面：

                • 请求日志控制
                • 分布式权重协调
                • 专家分布记录
                • 远端 transfer 信息查询

                到这里，代码导读就不只是在教你“系统怎么跑”，也开始教你“系统跑起来以后，维护者怎样通过正式控制面去看它、配它、协同它”。
